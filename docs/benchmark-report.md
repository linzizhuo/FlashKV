# FlashKV Dict 渐进式 Rehash 性能测试报告

**测试日期**: 2026-06-19  
**测试环境**: Linux 6.17, gcc -O2  
**测试方法**: `tests/bench_dict.c`，单次操作级延迟采样 (n=100,000)

---

## 1. 测试设计

| 指标 | 说明 |
|------|------|
| Throughput | ops/s，全局吞吐基线 |
| P50 / P95 / P99 / max | 延迟分位数，衡量尾部延迟 |
| σ (标准差) | 延迟抖动，越小越稳定 |
| rehash 期间 vs 整体 | 拆分 rehash 样本，单独对比 |

对比三组：
- **渐进式 (dictRehashData)**: 每次操作顺带搬 1 个**有数据的桶**，跳过空桶
- **全量式**: rehash 触发后一次性搬完所有桶（模拟无渐进式）
- **空桶跳过对比**: 单独对比 `dictRehashStep`（搬槽位，含空桶）vs `dictRehashData`（跳过空桶）

---

## 3. 空桶跳过对比：`dictRehashStep` vs `dictRehashData`

**测试目的**：验证"跳过空桶"策略是否减少无效搬迁调用。

**测试方法**：
- 两个相同 dict（size=131072, used=100000，负载因子 0.76）
- 手动触发 rehash → ht[1].size = 262144
- Dict A 用 `dictRehashStep(d, 1)` 完成搬迁 — 每次搬 1 个槽位（含空桶）
- Dict B 用 `dictRehashData(d, 1)` 完成搬迁 — 每次搬 1 个有数据的桶（跳过空桶）
- 统计完成 rehash 所需的总操作次数

### 3.1 操作次数对比

| 方法 | 总调用次数 | 空桶调用 | 数据调用 | 浪费率 |
|------|-----------|---------|---------|--------|
| `dictRehashStep` | 131,071 | 61,186 | 69,885 | 46.7% |
| `dictRehashData` | 69,885 | 0 | 69,885 | 0% |

**Data 比 Step 少 61,186 次无效调用（46.7%）。**

### 3.2 总耗时对比 (N=100000)

| 方法 | 总耗时 | 单次平均 | 备注 |
|------|--------|---------|------|
| `dictRehashStep` | ~10–30 ms | 235 ns | 波动大，受尾事件影响 |
| `dictRehashData` | ~8–21 ms | 301 ns | 单次稍贵，总调用少 |

> ⚠️ 耗时比受系统噪声影响大（实测 0.91× ~ 1.47×），纯 CPU 差异约 +10%（来自空桶脏写开销）。不要过度解读单次跑出的加速比。

### 3.3 纯空槽位开销：微基准

```
扫描 65536 槽 × 10000 轮：
  Read+dirty write:  0.39 ns/slot   (dictRehashStep 方式)
  Read only:         0.35 ns/slot   (dictRehashData 方式)
  → 空槽位 dirty write 额外 +10.8%
```

折算到 Hash 表搬迁场景：Step 比 Data 多 ~5% 左右的纯 CPU 开销。

### 3.4 延迟分布 (N=100000)

| 指标 | dictRehashStep | dictRehashData |
|------|---------------|----------------|
| P50 | 122 ns | 174 ns |
| P95 | 435 ns | 432 ns |
| P99 | 669 ns | 669 ns |
| **max** | **986 μs** | **138 μs** |
| σ | 4,254 ns | 1,028 ns |

Step 的 max/σ 更高主要因为调用次数多 (131K vs 70K)，更容易撞上系统调度的尾事件，**不是算法本身的性能问题**。

### 3.5 关键结论

```
dictRehashStep  → 需要 ~size  次操作完成 rehash（含大量空桶）
dictRehashData  → 仅需 ~used  次操作完成 rehash（每次都有进度）

→ 纯 CPU 收益有限（~5–10%），真正的价值在操作语义：
  ① 每次顺带搬迁保证有实际进度，rehash 更快完成
  ② 完成 rehash 需要的操作数从 size 降到 used（减少 46.7%）
  ③ 避免了空桶上的无谓 dirty cache line 写回
```

## 4. 结论

```
渐进式 rehash:          max=356μs   σ=2079ns   — 延迟均匀，无毛刺
全量式 rehash:          max=1675μs  σ=7220ns   — 周期性尾延迟爆炸
空桶跳过 (dictRehashData): 操作数 -46.7%, 保证每次操作有实质搬迁进度

→ 渐进式把 1.6ms 的周期性毛刺平摊为每次 ~300ns 的均匀开销
→ P99 波动不超过 3μs，用户完全无感
→ 跳过空桶让 rehash 在更少操作数内完成 (used 次 vs size 次)
```

核心结论：
1. **渐进式 rehash 消除了 rehash 触发时的延迟毛刺**，将尾延迟降低了 4.7 倍，同时标准差降低了 3.5 倍。
2. **`dictRehashData` 跳过空桶**，减少 46.7% 无效搬迁调用（纯 CPU 收益 ~5–10%），核心价值在于**操作语义**：每次顺带搬迁保证有实际进度，rehash 完成更快、更可预测。
