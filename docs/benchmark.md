# FlashKV Benchmark 报告

**日期**: 2026-07-01
**测试工具**: `redis-benchmark` (Redis 官方)
**测试环境**: Linux 5.15, GCC, 单机 loopback

---

## 测试方法

- 每次测试跑 **5 轮取平均值**，消除单次波动
- 每轮 50 并发连接，100,000~200,000 次请求
- 同一台机器上先后测试 FlashKV 和 Redis 6.0.16
- 脚本：[scripts/bench_compare.sh](../scripts/bench_compare.sh)

## 结果总览

| 场景 | **FlashKV** | **Redis 6.0** | FlashKV vs Redis |
|------|------------|--------------|:---:|
| `SET` P=1 | **62,268** ops/s | 60,415 ops/s | **+3.1%** |
| `GET` P=1 | **62,565** ops/s | 56,178 ops/s | **+11.4%** |
| `SET` P=16 | **698,174** ops/s | 660,333 ops/s | **+5.7%** |
| `GET` P=16 | 639,312 ops/s | **743,172** ops/s | -14.0% |
| `SET` P=64 | **1,428,959** ops/s | 1,120,648 ops/s | **+27.5%** |
| `GET` P=64 | **1,404,633** ops/s | 1,326,484 ops/s | **+5.9%** |

**6 项中 FlashKV 胜出 5 项。**

---

## 分析

### 为什么 FlashKV 单请求更快？

1. **无持久化开销**：没有 RDB 快照、AOF 日志，纯内存操作
2. **无复制开销**：没有主从同步带来的额外路径
3. **无多 DB 锁竞争**：虽然支持 16 个 DB，但访问是串行的
4. **轻量级协议解析**：零拷贝 RESP 解析器

### 为什么 P=16 GET 输给 Redis？

Redis 在中等 pipeline 场景下的批量读回路径经过多年打磨，包括：

- `writev` 聚合并发写
- 更优的 epoll 事件合并策略
- jemalloc 的内存分配优化

FlashKV 在 GET P=16 场景有提升空间，当前的批量回写路径（`handleRead → handleWrite → 尾递归`）在 16 条命令的窗口下额外触发了一次 epoll 往返。

### 为什么 P=64 SET 优势最大？

FlashKV 的 `MAX_PIPELINE_BATCH=64` 在这个窗口下恰好一次 `handleRead` 处理完毕，中间不切事件循环。配合 `TCP_NODELAY`，批量写入效率很高。Redis 在高 pipeline 下受限于其更复杂的内部调度（过期扫描、rehash 竞争等）。

---

## Pipeline 上限

FlashKV 设置 `MAX_PIPELINE_BATCH=64`，原因：

- **调度公平性**：单次 `handleRead` 处理约 1ms 的工作量，不影响其他连接的响应
- **内存控制**：防止 wbuf 在未 flush 前膨胀过大
- **值的选择**：16→64 的改动使 P=64 吞吐从 crash 提升至 140万+ ops/s
